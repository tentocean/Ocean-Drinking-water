## 1. ระบบนี้คืออะไร

ระบบจัดการลูกค้าและสรุปยอดขายรายวันของธุรกิจส่งน้ำดื่ม (รุ่งเรืองน้ำดื่ม จ.ระยอง)
ใช้งานจริงอยู่ — มีข้อมูลลูกค้าและยอดขายจริงในระบบ

**ผู้ใช้ 2 กลุ่ม:**
- **เจ้าของ/แอดมิน** — เห็นภาพรวมทุกสาย, ค่าใช้จ่ายโรงงาน, งบกำไรขาดทุน (ต้อง login)
- **พนักงานส่งน้ำ (5 สาย)** — บันทึกการขาย, เก็บเงิน, ดูลูกค้าสายตัวเอง (ไม่ต้อง login — แตะเลือกชื่อ)

**ฟีเจอร์หลัก:** บันทึกการขาย · จัดการลูกค้า · เก็บหนี้ค้าง · ออกใบกำกับภาษี/ใบเสร็จ (A4) · สรุปยอดรายวัน-เดือน-ปี · ค่าแรง&ค่าคอมมิชชั่น · ค่าใช้จ่ายโรงงาน · งบกำไรขาดทุน

## 4. เทคโนโลยีที่ใช้ (สำคัญ — ไม่ใช่ React project ปกติ)

**ไม่มี build step ไม่มี npm install** เป็นเว็บ static ล้วนๆ

ไฟล์ `.dc.html` เป็นฟอร์แมตของ Claude Design ประกอบด้วย 2 ส่วนในไฟล์เดียว:

1. **`<x-dc>` = เทมเพลต HTML** ใช้ `{{ ชื่อตัวแปร }}` ผูกข้อมูล และมี `<sc-if>` / `<sc-for>` แทน if/loop
2. **`<script data-dc-script>` = ตรรกะ** เป็น `class Component extends DCLogic` (React component)
   - `state = {...}` — ข้อมูลในหน่วยความจำ
   - `renderVals()` — คืน object ที่ชื่อ key ตรงกับ `{{ }}` ในเทมเพลต (นี่คือจุดเชื่อม)
   - method อื่นๆ — การคำนวณ/บันทึกข้อมูล

`support.js` จะอ่านเทมเพลต + โหลด React 18.3.1 จาก CDN (unpkg) แล้ว render ให้เอง
Supabase JS v2 โหลดจาก CDN (jsDelivr) ใน `<helmet>`

> **แก้ UI** → แก้ในส่วน `<x-dc>`
> **แก้ตรรกะ/การคำนวณ** → แก้ใน `<script data-dc-script>`
> **เพิ่มตัวแปรใหม่ให้เทมเพลต** → เพิ่ม key ใน `renderVals()`

## 5. ฐานข้อมูล (Supabase Postgres)

| ตาราง | เก็บอะไร |
|---|---|
| `employees` | พนักงาน 5 สาย (id เป็น text: `e1`–`e5`) |
| `profiles` | เชื่อม auth user → role (มีแถวเดียว = เจ้าของ) |
| `customers` | ลูกค้า (id เป็น uuid) — มี `name` (ชื่อสั้นไว้ค้นหา) และ `invoice_name` (ชื่อบริษัทเต็มสำหรับออกใบกำกับภาษี) |
| `sales` | การขาย — **1 แถว = 1 บรรทัดสินค้า** (`group_id` ใช้จับกลุ่มบิลเดียวกัน) |
| `expenses` | ค่าใช้จ่ายรายสาย (น้ำมัน/ทางด่วน/ฯลฯ) |
| `factory_expenses` | ต้นทุนโรงงาน (เจ้าของเท่านั้น) |
| storage bucket `slips` | รูปสลิปโอนเงิน |

Schema ทั้งหมดอยู่ใน `supabase/schema.sql` — **รันซ้ำได้ปลอดภัย** (drop-then-create policy, seed มี guard กันข้อมูลซ้ำ)

## 6. ⚠️ เรื่องความปลอดภัยที่ต้องรู้ (สำคัญที่สุดในเอกสารนี้)

**มีแค่เจ้าของที่ login จริง** พนักงานไม่ต้อง login — แค่แตะเลือกชื่อตัวเอง

ผลที่ตามมา: ฐานข้อมูลไม่มีทางแยกแยะว่าใครเป็นใครในฝั่งพนักงาน ดังนั้น RLS ของตาราง
`customers` / `sales` / `expenses` และ bucket `slips` จึง **เปิดให้อ่าน-เขียนได้กับทุกคนที่มี anon key**

แปลว่า **ใครก็ตามที่มีลิงก์เว็บนี้ สามารถดู/แก้ข้อมูลลูกค้าและยอดขายของทุกสายได้** —
การเลือกชื่อพนักงานเป็นแค่การ "ติดป้าย" ว่าข้อมูลเป็นของสายไหน **ไม่ใช่กำแพงความปลอดภัย**

- ✅ **ตั้งใจให้เป็นแบบนี้** เพื่อให้พนักงานใช้งานง่าย ไม่ต้องจำรหัส (เจ้าของเลือกเอง)
- ⚠️ **ห้ามแชร์ลิงก์เว็บสาธารณะ** — ลิงก์คือความลับตัวจริง
- ✅ `factory_expenses` และ `profiles` ยังจำกัดเฉพาะเจ้าของที่ login แล้วเท่านั้น

**anon key ที่ฝังในไฟล์ HTML เป็นเรื่องปกติ** (ออกแบบมาให้เปิดเผยได้ ปกติ RLS เป็นตัวกั้น)
แต่เพราะ RLS เปิดกว้างตามข้างบน ความปลอดภัยจริงจึงอยู่ที่ "ไม่มีใครรู้ลิงก์"

> **ถ้าวันหนึ่งต้องการความปลอดภัยจริงจัง** ต้องสร้าง login ให้พนักงานทุกคน แล้วเปลี่ยน RLS
> ให้ผูกกับ `profiles.employee_id` (โค้ดเดิมเคยเป็นแบบนี้ ดู git history commit `057f788`)

## 7. ตรรกะธุรกิจที่ควรรู้

- **`payCalc()`** — สูตรค่าแรง+ค่าคอมมิชชั่น **มีเงื่อนไขพิเศษ hardcode ไว้ว่า `empId === 'e1'` (แจ็ค) ใช้สูตรคนละแบบกับคนอื่น** ถ้าเปลี่ยนตัวพนักงานต้องระวังจุดนี้
- **ค่าแรง/ค่าคอม ถูกซ่อนจากมุมมองพนักงาน** (แสดงเป็น `—`) เจ้าของเห็นตัวเลขจริง — ซ่อนแค่การแสดงผล การคำนวณกำไรสุทธิยังใช้ค่าจริง
- **`TYPES`** (ประเภทน้ำ + ราคา), **`EXPCATS`** (หมวดค่าใช้จ่าย), **`FCATS`** (หมวดต้นทุนโรงงาน), **`SELLER`** (ข้อมูลผู้ขายบนใบกำกับ) — **hardcode ในสคริปต์ ไม่ได้อยู่ใน DB** ถ้าจะเพิ่มประเภทน้ำ/ราคา ต้องแก้ในไฟล์
- **ลบลูกค้า** — ลบได้เฉพาะลูกค้าที่ยังไม่มีประวัติการขาย (กันยอดขายเก่าหาย) ถ้ามีประวัติจะขึ้นข้อความปฏิเสธ

## 8. ใบกำกับภาษี / ใบเสร็จ

- เอกสารคือ element **`.tax-invoice`** (เป็น **class ไม่ใช่ id** เพราะพิมพ์ได้หลายแผ่น) — ดีไซน์ **มินิมอล ขาว-ดำ ไม่มีพื้นสี** (ประหยัดหมึก + ไม่ต้องพึ่ง "Print backgrounds")
- รองรับ 5 ประเภทเอกสาร (ใบกำกับภาษี, ใบส่งสินค้า, ใบเสร็จ, ใบวางบิล, ใบแจ้งหนี้) เลือกจาก dropdown
- **ต้นฉบับ + สำเนา:** ประเภท `taxreceipt` และ `delivery` พิมพ์ **2 แผ่น** (ต้นฉบับ→หน้า1, สำเนา→หน้า2) ประเภทอื่นแผ่นเดียว
  - ควบคุมด้วย `inv.copies` ใน `renderVals()` (array ของ `{label}`) → เทมเพลตวนด้วย `<sc-for list="{{ inv.copies }}">` ครอบทั้งใบ
  - ป้าย `(ต้นฉบับ)`/`(สำเนา)` อยู่บรรทัด docSub ผ่าน `{{ copy.label }}`
  - แต่ละสำเนาขึ้นหน้าใหม่ด้วย CSS `.tax-invoice + .tax-invoice{ break-before:page }`
- **ชื่อบนใบ** = `cust.invoiceName || cust.name` (ถ้าลูกค้ากรอกช่อง "ชื่อออกใบกำกับภาษี" จะใช้อันนั้น ไม่งั้นใช้ชื่อปกติ)
- **ซ่อนอัตโนมัติเมื่อไม่มีข้อมูล:** บรรทัด "โทร." ซ่อนถ้าลูกค้าไม่มีเบอร์ (`custHasPhone`), บรรทัดเลขภาษีซ่อนถ้าไม่มี (`custHasTax`)
- **ที่อยู่ผู้ขาย** แยก 2 บรรทัดผ่าน `SELLER.address` + `SELLER.address2` (ให้ ต./อ./จ. อยู่บรรทัดเดียวกัน)
- **ช่วงเวลาแบบ "กำหนดเอง"** default = **25 เดือนก่อน → 24 เดือนปัจจุบัน** (รอบบิล) ดู `defaultBillingRange()`
- **การพิมพ์** (CSS ใน `@media print` ด้านบนไฟล์): `.app-root > *:not(.print-invoice){ display:none }` ซ่อนแอปทั้งหมดยกเว้น modal ใบ แล้วให้ `.tax-invoice` ไหลเต็มพื้นที่พิมพ์
- ⚠️ **ระยะขอบใช้ `@page{ margin:16mm 14mm }`** (ไม่ใช่ padding ของ element) และ `.tax-invoice{ height:auto }` — **อย่าใส่ `height:297mm`** เพราะสูงเท่าหน้าเป๊ะจะล้นไปหน้าถัดไป

## 9. รองรับมือถือ (สำคัญ — พนักงานใช้มือถือเป็นหลัก)

CSS มือถืออยู่ใน `@media (max-width:640px)` ด้านบนไฟล์ ใช้คลาสชื่อขึ้นต้น `m-*`:

| คลาส | ทำอะไร |
|---|---|
| `m-tabs` | แท็บเต็มความกว้าง เห็นครบ ไม่ต้องเลื่อน |
| `m-cust` | แท็บลูกค้าเรียงคอลัมน์เดียว (ประวัติอยู่ใต้ช่องค้นหา) |
| `m-hist` | ตารางประวัติ → การ์ดต่อรายการ (ใช้ `data-label` ทำป้ายกำกับ) |
| `m-saleline` | แถวบันทึกการขายจัดใหม่ให้พอดีจอ |
| `input,select,textarea` = 16px | กัน iOS ซูมเองตอนแตะช่องกรอก |

> เนื่องจากทั้งไฟล์ใช้ **inline style** การ override บนมือถือต้องใช้ `!important` ผ่านคลาส `m-*` เหล่านี้
> **เดสก์ท็อปไม่ได้รับผลกระทบ** — กฎทั้งหมดอยู่ใน media query

## 12. ข้อควรระวัง / กับดักที่เคยเจอ

- **`support.js` เป็นไฟล์ generated — ห้ามแก้** (จะโดนทับถ้า regenerate)
- **เปลี่ยนรูปในโฟลเดอร์ assets แล้วไม่ขึ้น** → เบราว์เซอร์/Cloudflare cache ไว้ ต้องต่อ `?v=2` (โลโก้ตอนนี้ใช้ `logo.png?v=2` อยู่)
- **schema.sql รันซ้ำได้** แต่ลำดับสำคัญ — ต้อง drop policy ก่อน drop function ไม่งั้น error
- **เพิ่มคอลัมน์ DB ต้องรัน SQL เองก่อน** — โค้ดที่ insert คอลัมน์ที่ DB ยังไม่มีจะ error เงียบ (เช่น `invoice_name` เคยทำให้ "เพิ่มลูกค้าไม่ได้") schema.sql มี `alter table ... add column if not exists` ให้แล้ว
- **พิมพ์แล้วได้หน้าว่างแผ่น 2** — เคยเกิดจาก `.app-root{ min-height:100vh }` เพราะตอนพิมพ์ Safari คิด `vh` จากจอ ไม่ใช่หน้ากระดาษ → แก้ด้วย `.app-root{ min-height:0 }` ใน `@media print` (อย่าลบทิ้ง)
- **ข้อมูลตัวอย่างในเดโมเดิมเป็นข้อมูลสุ่ม** ไม่ได้ย้ายเข้า DB — ยอดขาย/ค่าใช้จ่ายเริ่มจากศูนย์ ส่วนพนักงาน 5 คน + ลูกค้า 16 รายถูก seed ไว้
- **ไฟล์แอปชื่อภาษาไทย** เวลาใช้ใน URL ต้อง encode (`%E0%B8%A3...`) — `index.html` ช่วย redirect ให้แล้ว

## 13. สิ่งที่ยังไม่ได้ทำ / ข้อเสนอแนะ

- **ความปลอดภัยฝั่งพนักงาน** — ดูข้อ 6 ถ้าธุรกิจโตขึ้นควรทำ login รายคน
- **ซ่อนลูกค้า (archive)** — ตอนนี้ลบได้เฉพาะลูกค้าที่ไม่มีประวัติ ถ้าลูกค้าเลิกซื้อแต่มีประวัติแล้ว จะลบไม่ได้เลย ควรทำ "ซ่อน" แทนลบ
- **ข้อความเกณฑ์ค่าแรงยังโชว์อัตราให้พนักงานเห็น** (เช่น "ถังละ ฿1") แม้จะซ่อนยอดเงินแล้ว ถ้าต้องการซ่อนด้วยต้องแก้ `payCalc()` ตรง `sub` และ `criteria`
- **ยังไม่มี test อัตโนมัติ** — ทดสอบด้วยการเปิดเว็บจริงเท่านั้น
- **ราคา/ประเภทน้ำ hardcode** ถ้าต้องเปลี่ยนราคาบ่อย ควรย้ายไปเก็บใน DB
